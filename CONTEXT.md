# Pi Environment

This context describes the local Pi agent environment and workflow-extension concepts used to orchestrate AI-assisted engineering work.

## Language

**Skill Routing**:
Choosing extra skills for a ticket before launching an implementation or review agent. Skill routing is hybrid and two-stage: labels provide deterministic first-pass signals, issue/body analysis can add skills when concrete evidence supports them, and the worker may adjust the proposed pack after repo exploration when it finds new evidence. Parent orchestration does not need to recompute routing mid-run. Worker adjustment may add any available registry skill, including repo-defined skills, but must not invent unregistered skill IDs or browse arbitrary skill files outside the registry. Skill routing should have a read-only dry-run/explain command named `/matt-route-skills <issue>` for tuning routes before wiring or launching automation. V1 dry-run routing requires a GitHub issue target rather than arbitrary text or current diff, and always validates repo route config and the active skill registry before reporting packs. Skill routing should be implemented as reusable routing logic that commands consume, not embedded only in prompt strings.
_Avoid_: skill injection, skill magic, auto-loading

**Skill Routing Module**:
A reusable code module that loads registry/config, validates route health, matches issue evidence, builds agent-specific skill packs, detects overflow, and formats routing explanations. `/matt-route-skills`, `/matt-slice`, `/matt-afk`, and `/matt-auto` should consume this shared module. Core routing should return structured data, with separate formatters for dry-run summaries, worker/review prompt contracts, and slice-time issue hints. Child-agent contracts should include selected skill IDs, absolute `SKILL.md` paths, and evidence-backed rationale; child agents should read selected skill files as mandatory upfront guidance before acting instead of receiving full skill contents inline. Child agents should output only a compact `Skill adjustments` note when they change the parent-proposed pack, or `none` when unchanged; skill routing is guidance uplift rather than an audit checklist. Commit messages and closeout comments should describe actual changes and verification, not name the skills used. `/matt-auto` route results should stay in memory and appear only in the compact final loop log or stop reason; auto should not persist routing internals to repo files or issue comments. Skill routing does not affect `/matt-auto` queue ordering in v1; it only changes how selected issues are worked or stops when routing exposes missing skills or overly broad slices. V1 should not infer repository tech-stack prerequisites or route-level `requires`; route authors are responsible for configuring routes that fit their repo. The implementation should be split into a `skill-routing/` module from the start rather than embedded in `index.ts`, with first-class unit tests for validation, merging, matching, confidence promotion, overflow, scaffold shape, and pack formatting. `/matt-route-skills` should show selected packs plus a compact considered/skipped section for partial, low-confidence, disabled, or otherwise relevant routes, not every unmatched route.
_Avoid_: prompt-only router, duplicated command-specific routing logic, brittle prompt-text assertions, inlined full skill dump, audit-style skill policing, skill-name closeout, route-log issue spam, skill-based queue priority, inferred tech-stack gating, monolithic index.ts router, untested config logic, noisy route debugger dump

**Worker Skill Pack**:
The small selected set of allowlisted skills given to the implementation agent session that picks up a ticket. A worker skill pack should be explainable and capped so the agent gets targeted guidance instead of broad prompt noise. Most implementation-relevant skills are expected to be compatible with review too.
_Avoid_: all relevant skills, full skill dump

**Review Skill Pack**:
The small selected set of allowlisted skills given to the review agent session for a ticket. Most route-selected skills may appear in both worker and review packs, but some skills are review-only when their purpose is auditing rather than implementation, such as security review.
_Avoid_: reviewer prompt dump, one-size-fits-all review lens

**Agent Compatibility**:
The explicit set of agent roles a routed skill may be sent to. Compatibility is modeled as a list of roles such as `worker` and `review`, not as a `both` shortcut, so future agent roles can be added without changing the model.
_Avoid_: both, universal compatibility, hardcoded two-agent assumptions

**Baseline Skill Pack**:
The mandatory default skill pack for Matt workflow automation before ticket-specific routing is applied. The worker baseline includes `implement` and `tdd`; route-selected skills add specialization on top. Baseline skills should be shown explicitly in generated worker/review contracts rather than hidden as implicit phase behavior. Repo `disabledSkills` affects routed skills only and does not remove baseline skills in v1.
_Avoid_: empty baseline, phase prompt replacement

**Pack Cap**:
The configurable maximum number of routed, ticket-specific skills included in a skill pack after deduplicating by skill ID, not counting baseline skills. Atomic issue slices should normally warrant only three to four routed skills; needing more is evidence that the issue may be too broad or should be split further. Defaults are `workerMaxRoutedSkills: 3` and `reviewMaxRoutedSkills: 4`, with repo config allowed to override without hard upper bounds; configured limits must still be positive integers, and `0` is not a disable mechanism. If high-confidence routed skills exceed the active cap, Matt auto mode should stop before implementation and recommend human re-slicing instead of silently dropping high-confidence skills. Medium-confidence overflow does not stop automation; include high-confidence skills first, then medium-confidence skills until the cap, and list the rest as considered/skipped due to cap. Skill packs sort baseline skills first, then routed skills by confidence, preserving active route order within the same confidence.
_Avoid_: prompt bloat, broad slice, hard limit bounds

**Worker Skill Allowlist**:
The curated set of skills that Matt workflow automation may route into AFK implementation or review sessions. The allowlist protects auto loops from accidentally using broad, interactive, destructive, or account-mutating skills. Extension defaults may be narrowed or extended by repo-local route config.
_Avoid_: installed skills, global skill library

**Repo Skill Route Config**:
A project-owned `.pi/matt-skill-routes.json` file layered on top of the extension's default skill routes. It lets a repo map its own labels, paths, and issue language to allowlisted worker or review skills without changing the Matt workflow extension. Repo skill routes select skill IDs with evidence-backed rationale, not arbitrary prompt snippets or binding policy text, and may define repo-owned skill registry entries. V1 uses strict JSON rather than JSONC, YAML, or executable TypeScript and requires `version: 1` so future config migrations can hard-stop clearly on missing or unsupported versions. Repo config is additive: it adds skills and routes, then applies explicit `disabledRoutes` and `disabledSkills`; v1 does not deep-patch default route objects. V1 route matching uses only positive `labels`, `title`, `body`, and `paths` fields; `labels` use explicit normalized matching against the current route shape, not broad default alias vocabularies; `title` and `body` use plain case-insensitive substring tokens, not regex. V1 supports route-level confidence only, defaulting to `medium`, with router-owned promotion rules for strong signals or multiple matches. Invalid repo config is a hard stop only for commands that use skill routing, such as `/matt-route-skills`, `/matt-slice` skill hints, `/matt-afk`, and `/matt-auto`; unrelated planning/status commands should not be blocked. Routing-aware commands validate route config before fetching GitHub issues or doing target-specific work. V1 includes a scaffold command named `/matt-init-skill-routes` to create only the initial `.pi/matt-skill-routes.json` without overwriting existing config; it should include all top-level sections for discoverability, including explicit default top-level `limits`, and should not create example skill files or directories. V1 does not support per-token confidence or per-route negative evidence blocks such as `unless`.
_Avoid_: global config, hardcoded project rules, executable config, deep route override, per-route unless rules, regex route matching, per-token confidence, broad label alias vocabulary, unversioned config

**Evidence Signal**:
A label, issue title/body phrase, acceptance criterion, parent PRD detail, explicit file-path hint, touched-file path, discovered code detail, or slice-time agent skill hint used to justify selecting a skill. Conflicting signals are additive and weighted: child acceptance criteria outrank child labels/body, child file paths outrank inherited parent PRD context, and repo-local defaults are weakest. Route fields use OR matching for initial selection, but confidence depends on signal strength and reinforcement: broad labels alone are usually medium, strong acceptance/body phrases can be high, and multiple medium signals can combine into high. Evidence produces simple `high`, `medium`, or `low` route confidence: high is automatically selected, medium is selected when the pack cap allows, and low is considered but skipped unless reinforced. V1 confidence promotion is generic, not route-specific: route config has no per-route promotion knobs, and the router owns simple promotion rules such as multiple matching evidence fields raising confidence. When multiple routes select the same skill, route results deduplicate by skill ID, keep the highest confidence, and merge all route IDs/evidence for explanation. Milestone titles or descriptions are not routing evidence because milestones are broad delivery arcs, not ticket-specific implementation signals. Slice-time hints are low-authority evidence; auto mode recomputes routing from final child issue state before implementation.
_Avoid_: guess, vibe, heuristic without evidence, milestone-as-skill-signal, route-specific promotion knobs

**Agent Skill Hint**:
Durable skill-routing metadata written to child issues during slicing. Agent skill hints should have both a visible Markdown summary for humans and machine-readable metadata for stable automation parsing, but remain low-authority evidence that auto mode recomputes before implementation. Machine-readable hints include resolved skill IDs plus route IDs as diagnostic provenance. `/matt-slice` should hard stop before creating child issues when routing config is invalid, rather than creating issues with missing or misleading skill hints. V1 has no `--no-skill-hints` bypass; fixing route config is the intended path.
_Avoid_: hidden-only metadata, prose-only metadata, binding route decision, skill-hint bypass

**Missing Routed Skill**:
A skill selected by active route config that is unavailable in the current Pi session. Missing routed skills are hard stop conditions for Matt auto/AFK automation; the loop should not silently skip, fallback, or claim the skill was used.
_Avoid_: graceful fallback, unavailable warning only, hallucinated skill use

**Skill Registry**:
The active catalog of routeable skill IDs, compatibility, safety, and resolver metadata. It starts with extension-owned defaults and may be extended by repo-local config for project-specific skills. Extension defaults may include known high-value non-Matt local skills, such as accessibility, React performance, testing, observability, and security-review lenses, but strict availability validation still applies. V1 default routes should be small, high-signal, enabled automatically for routing-aware commands, and defined in TypeScript code for type-checking; repos add domain-specific breadth or disable defaults through repo-local JSON config. The registry should not hardcode absolute paths in persisted config; it should store stable IDs plus source/relative-path metadata and compute availability from the current Pi session, extension root, or repo root at runtime. Repo-defined skill resolvers are repo-relative local `SKILL.md` paths only in v1; any repo-internal path is allowed, with `.pi/skills/<id>/SKILL.md` as the recommended convention. No remote URLs or shell-command resolvers.
_Avoid_: extension-only skill catalog, persisted absolute paths, runtime-only opaque lookup, remote skill resolver, broad default route set
